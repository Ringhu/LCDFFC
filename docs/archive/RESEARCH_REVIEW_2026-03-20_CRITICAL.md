## 对你的问题逐点回答

## 1. 关于 5.1：为什么“不重训低层系统”？

### 当前文档的问题

`docs/project_progress_explainer_2026-03-20.md` 里把“不重训低层系统”写成了方法设定，但没有解释清楚这是：

- 一个**论文问题收缩策略**
- 不是一个“低层已经最优”的事实判断

### 更准确的解释

不重训低层系统的真正原因是：

> 为了把论文问题收缩成“高层目标适配”而不是“高层 + 低层一起重新学”。

如果同时重训低层预测器和控制器，那么一旦结果变好，你就无法判断收益到底来自：

- 更强的高层路由
- 更强的预测器
- 更强的控制器
- 还是三者耦合后的共同作用

所以“冻结低层”是一种**变量隔离策略**，不是一种“低层已经最优”的证明。

### 如果重训会怎么样

如果重训低层，有两种可能：

1. 结果变好
   - 但论文主张会变模糊，因为你不再是在证明“高层目标适配”
2. 结果不变或变差
   - 仍然无法说明高层机制本身到底有没有价值

这会把当前 focused thesis 重新拉回 broad idea，反而更难讲清楚。

### 怎么判断低层系统是不是最优

当前**根本不能**判断低层系统是不是最优。

目前最多只能说：

- 它是当前工程里已经打通并可复现的 backbone
- 它在当前设置下比 `RBC` 更好
- 它足够作为高层实验的平台

但它绝对不能写成：

- 最优 forecast backbone
- 最优 control backbone
- 已经充分代表低层上界

### 当前必须补的改进

如果你后面要让论文更可信，至少需要补下面一种低层比较：

- `GRU` vs 一个更强的时序 baseline
- `QP` vs 一个更强的控制 baseline

否则 reviewer 很容易说：

> 你看到的所谓高层收益，只是建立在一个并不强的低层系统上。

## 2. 关于 5.2：什么样的偏好切换才是“好的”？

### 当前协议的问题

当前 `preference-shift` 协议基本是：

- 把 episode 硬切成 4 段
- 每段分别对应 `cost / carbon / peak / reserve`

这作为 pilot protocol 是可以的，但作为论文最终协议，**太人工**。

### 什么样的偏好切换才是更好的

更好的偏好切换应满足 4 个条件：

1. **和真实运营语境有关**
   - 例如价格高峰、碳强度高峰、负荷高峰、备用风险上升
2. **切换有外部触发原因**
   - 而不是作者手工平均切 4 段
3. **切换时间和持续时间不固定**
   - 更接近真实运行
4. **高层指令可以从外部信息中解释出来**
   - 否则“语言条件”就只是人为标签

### 如何切换才更合理

更合理的切换方式应该是：

- 当价格高于某阈值时进入 `cost-first`
- 当碳强度升高时进入 `carbon-first`
- 当负荷预测高峰接近时进入 `peak-first`
- 当 `soc` 低且未来风险上升时进入 `reserve-first`

也就是说，偏好切换最好由**场景驱动**，而不是由作者手工等长切段。

### 这个如何作为 baseline

当前 preference-shift 里的 baseline 至少应该包括：

1. 单一固定控制器
   - 如 `fixed_reserve`
2. 简单规则路由
   - `heuristic_router`
3. 数值偏好向量路由
4. 当前文本路由
5. regime-wise best fixed 上界

当前这些 baseline 基本有了，但缺两样：

- 更真实的 event-driven shift protocol
- 多个 episode / 多个场景的统计结果

所以：

> baseline 的“种类”基本够，但 baseline 所在的协议还不够强。

- Chronos

最合理的解释只能是：

> `GRU` 是一个简单、稳定、便于先打通闭环的起点。

它适合作为：

- 原型验证 backbone
- 低成本 baseline

1. 补一个强但代价可控的经典时序 baseline

## 4. 关于 6.2：为什么用 QP Controller？其他控制方法可以吗？

### 当前真实状态

当前控制层是 `controllers/qp_controller.py`，本质上是一个凸优化的 receding-horizon controller。

它的优点是：

- 容易写目标函数
- 容易加约束
- 可解释
- 相对稳定

### 其他控制方法可以吗？

当然可以。

例如：

- heuristic / rule-based controller
- MPC（更一般的）
- challenge winner style optimization
- RL controller
- hybrid controller

### 为什么当前选 QP

当前选 QP 最合理的理由只能是：

> 它是一个适合快速验证 “forecast-then-control” 主线的、稳定且可解释的低层控制器。

它不等于：

- 最优控制器
- 唯一合理控制器
- 已经足够代表控制层上界

### 怎么判断它是不是最优

当前也**不能判断**。

要判断，至少要补控制层对比：

- `RBC`
- myopic heuristic
- QP / MPC
- challenge-style optimization baseline（如果能复现）

否则当前最多只能说：

- QP 是一个工作正常、可复现的低层控制 backbone

### 当前必须修正的文档口径

所以文档里不应该让人读出：

- “QP 是被证明最优的控制层”

更准确的说法应该是：

- “QP 是当前为了固定低层主循环而选择的可解释 backbone”

## 5. 关于 6.3：语言路由到底用到了什么方法？是不是 LLM + agent？

### 当前真实状态

这一点必须说得很直接：

> **当前并没有真正实现可运行的 LLM + agent 路由。**

原因是：

- `llm_router/router.py` 里的 `LLMRouter.route()` 仍然是 `NotImplementedError`
- 当前跑通的所谓“语言路由”都在 `llm_router/preference_routers.py`
- 这些实现本质上是：
  - regex / 模板 / 规则
  - 数值上下文调整
  - expert profile 选择与轻量混合

也就是说，当前 `text_v4` 虽然是“语言条件化”的，但它不是一个真实的 LLM inference system，更不是一个完整的 agent。

### 能不能做成 LLM + agent？

当然能，而且如果你的原始研究初衷不想丢，后面迟早要做。

但当前没有。

### 这对论文意味着什么

这意味着当前论文不能写成：

- “LLM agent for control”
- “基于大语言模型的智能体控制系统”

更稳妥的表述只能是：

- `language-conditioned high-level router`
- 或者更保守一点：
  `instruction-conditioned routing surrogate`

### 和其他方法怎么比

如果以后真的实现 LLM 路由，那么至少要比较：

1. heuristic router
2. numeric preference router
3. 当前 deterministic text router
4. 真正的 LLM router

否则你无法证明 LLM 的增量价值。

所以这部分当前最大的问题不是“效果还差一点”，而是：

> 目前根本还没有一个真实的 LLM baseline / method 被实现出来。

## 6. 关于第 8 节结果：为什么 regret 都在 0 附近？这是不是说明差异没意义？

### 这是一个非常关键的质疑，而且你是对的

当前这些值：

- `text_v4 = 0.876622`
- `fixed_reserve = 0.876931`
- `heuristic_router = 0.878355`

从数字上看，差异都非常小。

尤其 `v4` 相对 `fixed_reserve` 的优势只有：

- `0.000309`

这个幅度确实很弱。

### 为什么会这样

有两个原因：

1. 当前指标是一个**归一化后的复合分数**
   - 它本身会把差异压缩
2. 当前 protocol 可能还不够难
   - 导致很多方法都能拿到接近的分数

所以你看到 regret 接近 0，并不一定说明“方法都一样”，但它至少说明：

> 当前协议下的区分度还不够强，不能把这种极小优势直接当成强论文证据。

### 当前文档哪里写得不够严谨

像“`text_v4` 优于最佳单一固定控制器”这种话，严格来说没错，
但如果不补充“幅度非常小”，就容易误导。

更准确的写法应该是：

> `text_v4` 在当前协议下相对最佳单一固定控制器出现了稳定但幅度较小的优势，这更适合作为正向方向信号，而不是最终定论。

### 这对论文的真实影响

如果你现在拿着这点差距直接写论文主结果，reviewer 很可能会问：

- 有没有统计显著性？
- 多个 episode 是否仍成立？
- 协议是不是太简单？
- 这个增益在工程上是否有意义？

所以你的直觉是对的：

> 当前结果可以说明方向可能对，但还不够说明“已经做出一个很强的论文结果”。

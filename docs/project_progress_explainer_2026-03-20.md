# 从 0 开始看懂 LCDFFC 当前做到哪一步（2026-03-20）

## 1. 先用一句话说，这个项目到底在做什么

这个项目想解决的问题是：

**当未来的电价、碳排放强度、建筑负荷都会变化时，能不能先做时间序列预测，再根据预测结果控制建筑电池，从而让整体用电更便宜、更低碳、峰值更低？**

更进一步，这个项目还想回答：

**如果人的控制目标会变，比如一会儿更关心省钱，一会儿更关心削峰，一会儿更关心保留电池电量，那么能不能让一个语言条件化的高层路由器去动态调整控制目标，而不用每次重新训练整个控制系统？**

所以，这不是一个纯预测项目，也不是一个纯强化学习项目，而是一个：

**时间序列预测 + 优化控制 + 高层语言路由**

的组合研究。

---

## 2. 研究背景：为什么要做这件事

现实里的能源系统控制有两个难点：

1. **未来是不确定的**
   - 电价会变
   - 碳强度会变
   - 建筑负荷会变
   - 光伏出力会变

2. **控制目标不是永远固定的**
   - 有时更想省钱
   - 有时更想减少碳排放
   - 有时更想降低最大峰值负荷
   - 有时更想保留电池电量，防止后面出风险

这意味着，一个真正有用的系统不能只会“在固定目标下做控制”，而应该具备两种能力：

- 能根据时间序列预测来提前做准备
- 能根据高层目标变化来动态调整控制策略

这就是这个项目的研究背景。

---

## 3. 研究场景：为什么用 CityLearn

这个项目当前的主实验平台是 `CityLearn Challenge 2023`。

你可以把 `CityLearn` 理解成一个“建筑能源控制仿真平台”。

它提供：

- 多栋建筑的能耗场景
- 天气、电价、碳强度等时间序列
- 电池等可控资源
- 统一的仿真接口和评价指标

在这个项目里，`CityLearn` 的作用很重要，因为它不是随便一份表格数据，而是一个可以让控制器真实“做动作 -> 看后果 -> 拿指标”的环境。

所以这里的研究不是离线预测比赛，而是一个**闭环决策问题**。

---

## 4. 当前的研究任务是什么

如果把当前任务说得足够具体，就是：

**在 CityLearn 里，只控制建筑群的电池充放电，先用历史时间序列预测未来风险，再让控制器根据目标函数做优化决策，最后比较这种方法能不能优于简单基线。**

当前不是控制所有设备，而是聚焦在：

- `central_agent=True`
- `battery-only control`

也就是说，现在主要控制的是**电池**。

---

## 5. 当前真正的研究问题是什么

项目一开始的大方向很宽：

- 时间序列分析
- 决策
- LLM agent
- 多个 benchmark

但做到现在，研究问题已经被收缩成了一个更清楚、也更适合写论文的问题：

> **在不重训低层控制系统的前提下，语言条件化的高层路由器，能不能在偏好切换场景里优于单一固定权重控制器？**

这里有两个关键词：

### 5.1 不重训低层系统

低层的预测器和控制器先固定住。

也就是说，不是每次目标一变就重新训练一遍模型，而是希望用一个高层模块去调整目标和约束。

### 5.2 偏好切换场景

所谓“偏好切换”，可以理解成：

- 前一段更看重省钱
- 下一段更看重低碳
- 再下一段更看重削峰
- 后一段更看重保留电池余量

项目现在的论文主问题已经不是“能不能控制”，而是：

**在这些偏好切换下，语言条件化高层路由到底有没有真实价值。**

---

## 6. 当前的方法是怎么工作的

当前方法可以分成三层看。

### 6.1 第一层：时间序列预测层

这里用的是一个 `GRU` 预测器。

它做的事情很简单：

- 输入最近一段历史观测
- 预测未来若干步的关键量

当前重点是让预测结果服务后面的控制，而不是只追求预测误差好看。

### 6.2 第二层：优化控制层

这里用的是一个 `QP Controller`。

你可以把它理解成：

- 给它一个未来预测
- 给它一个目标函数
- 它会求一个当前应该怎么充电/放电的动作

它不是 RL 直接输出动作，而是一个更稳定、更可解释的优化控制器。

### 6.3 第三层：高层语言路由层

这是当前研究里最像“论文贡献”的部分。

它不直接输出底层连续动作，而是做：

- 识别当前高层目标
- 选择或调整控制器的权重和约束

当前实验结论表明，最好的版本不是“自由生成连续权重”，而是：

**把语言映射到已经被验证有效的一组 expert profiles，再在这些 experts 上做受限选择与轻量混合。**

当前这个 best 版本叫：

- `text_v4`

---

## 7. 现在做到什么程度了

这是最关键的一节。

### 7.1 已经完成的基础能力

已经完成并且能跑通的部分包括：

- 从 `CityLearn` 提取数据
- 构建时间序列数据集
- 训练 `GRU` 预测器
- 运行 `QP` 控制器
- 跑 `RBC` 基线
- 跑 `forecast + QP` 的闭环评估
- 跑高层路由实验
- 跑误差分析和 fallback 鲁棒性实验

也就是说，项目已经不再是一个空框架，而是有一条完整可运行的实验主线。

### 7.2 已经验证过的关键结论

#### 结论 1：低层主闭环已经成立

当前 `learned forecast + QP` 已经在本地验证里优于 `RBC`。

这说明项目最底层的“先预测、再控制”思路已经不是空想，而是真能跑出正向结果。

#### 结论 2：高层语言路由已经有正信号

当前最好的文本路由版本是 `text_v4`。

它相对最佳单一固定控制器 `fixed_reserve` 有稳定优势。

这说明“语言条件化高层路由优于单一固定权重控制器”这个主张，已经有实验支撑。

#### 结论 3：误差来源已经被拆清楚

项目现在已经知道问题主要出在哪：

- `reserve` 是第一主敏感点
- `carbon` 是第二主敏感点

这意味着你现在不是只知道“模型还不够好”，而是知道“还差在哪一类场景”。

#### 结论 4：fallback 已经被证明有用

在专门构造的失效场景下，fallback 不是摆设。

它已经被证明可以：

- 保护 `peak`
- 保护 `ramping`
- 在 reserve 相关错误下恢复安全性

所以安全性分析已经不只是设想，而是有真实结果。

## 8. 当前最好的实验结果是什么

当前最重要的一组结果是 `preference-shift` 评测中的汇总分数。

关键方法的结果如下：

| 方法 | avg_preference_score | avg_regret_to_best_fixed | avg_regret_to_best_single_fixed |
|---|---:|---:|---:|
| fixed_reserve | 0.876931 | 0.001224 | 0.000000 |
| heuristic_router | 0.878355 | 0.002647 | 0.001424 |
| text_router_v2 | 0.876864 | 0.001157 | -0.000067 |
| text_router_v4 | 0.876622 | 0.000914 | -0.000309 |
| text_router_v5 | 0.876862 | 0.001154 | -0.000070 |
| text_router_v6 | 0.876622 | 0.000914 | -0.000309 |
| text_router_v7 | 0.876622 | 0.000914 | -0.000309 |

这里不用死记数字，只需要记住三件事：

1. 当前 best 是 `text_v4`
2. `text_v4` 优于最佳单一固定控制器
3. 后续 `v5 / v6 / v7` 都没有再超过 `v4`

这说明当前最优点已经比较稳定。

## 9. 当前还没做完的部分是什么

虽然项目已经做到了“可以开始写 paper”的阶段，但它还没有彻底闭环。

主要还差下面几块：

### 9.1 还没有追平更强上界

当前 best `text_v4` 还没有追平 `regime-wise best fixed upper bound`。

也就是说，你已经超过了“单一固定控制器”，但还没有超过“如果每一段都人工挑最优固定 expert”的理想上界。

### 9.2 还没有 OOD / transfer 证据

目前结果主要建立在当前 `CityLearn preference-shift` 协议上。

还没有额外证明：

- 分布变化时是否仍然有效
- 迁移到别的环境时是否仍然成立

### 9.3 原始大方向里的很多内容还没进入主论文闭环

比如：

- decision-focused learning
- uncertainty-aware full path
- Grid2Op 第二验证
- 更大范围的跨环境方法论文

这些不能说没价值，但现在还没进入当前这篇 focused paper 的闭环里。

## 13. 勘误与修正（基于 2026-03-20 批判性复审）

这份说明文档原本是按“帮助快速理解项目”来写的，因此有几处表述过于顺滑。基于后续复审，需要补充下面这些修正：

1. **5.1 中“低层系统先固定住”不等于“低层已经最优”**
   - 现在冻结低层，主要是为了隔离论文变量，先看高层目标适配有没有价值。
   - 当前不能把 `GRU + QP` 写成已证明的最优低层系统。

2. **6.1 中当前真正实现的预测模型只有 `GRU`**
   - `PatchTST`、`TSMixer`、Chronos、TimesFM、MOMENT 等都还没有在当前工程里真正实现并对比。
   - 所以现在最多只能说 `GRU` 是 working backbone，不能说它已经代表最佳时序预测层。

3. **6.2 中当前真正实现的低层控制器是 `QP Controller`**
   - 它的优点是稳定、可解释、易加约束。
   - 但当前还没有足够控制层对比，不能说 QP 已被证明优于其他控制方法。

4. **6.3 中当前“语言路由”不是实际的 LLM + agent 系统**
   - 当前 best `text_v4` 本质上是 deterministic text-conditioned expert selector。
   - `llm_router/router.py` 里的真实 `LLMRouter.route()` 还没有实现。
   - 因此当前更准确的说法是：`instruction-conditioned router`，而不是完整的 `LLM agent`。

5. **8 中当前结果虽然是正向信号，但幅度仍然偏小**
   - `text_v4` 相对 `fixed_reserve` 的优势存在，但数量级很小。
   - 这更像“方向正确的 pilot signal”，还不够作为强投稿版主结果直接使用。

6. **10 中“现在能不能写论文”需要更精确地理解**
   - 现在可以开始写内部论文骨架、paper plan、paper draft。
   - 但还不应该把当前状态理解成“已经接近投稿完成”。

如果后续要把这条主线真正写成更可信的论文，优先需要补的是：

- 更强的低层 baseline（预测层/控制层）
- 更真实的 preference-shift 协议
- 真实可运行的 LLM 路由实现，或者明确放弃 LLM claim
- 多 episode / OOD / transfer 类证据
